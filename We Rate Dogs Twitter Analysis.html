Archive Dataframe Explained (Twitter archive) columns:

  • tweet_id: the unique numerical identifier for each tweet
  • in_reply_to_status_id: if the represented tweet is a reply, this field will contain the integer representation of the original tweet ID
  • in_reply_to_user_id: if the represented tweet is a reply, this field will contain the integer representation of the original tweets author ID
  • timestamp: time when this Tweet was created
  • source: utility used to post the tweet, as an HTML-formatted string.
  • text: actual UTF-8 text of the status update
  • retweeted_status_id: if the represented tweet is a retweet, this field will contain the integer representation of the original tweets ID
  • retweeted_status_user_id: if the represented tweet is a retweet, this field will contain the integer representation of the original tweets author ID
  • retweeted_status_timestamp: date/time representation of retweet
  • expanded_urls: tweet URL
  • rating_numerator: numerator of the rating of a dog. Ratings are almost always greater than 10, a fun quirk of this particular Twitter Feed
  • rating_denominator: denominator of the rating of a dog. Note: ratings almost always have a denominator of 10
  • name: Dog's name
  • doggo: one of the 4 dog "stages"
  • floofer: one of the 4 dog "stages"
  • pupper: one of the 4 dog "stages"
  • puppo: one of the 4 dog "stages"

Look over individual dataframes to assess for a minimum of 8 quality issues and 2 tidiness issues

Quality issues pertain to content. There are four characteristics of quality data:

  • Completeness: Are any data values missing? Are there specific rows, columns, or cells missing?
  • Validity: Recorded data with validity issues may not conform to a defined schema. (e.g. negative value for height is impossible)
  • Accuracy: inaccurate data is wrong data that is technically valid. Inaccurate date follows the defined schema, but is incorrect. (e.g. a patient's weight that is 5 lbs too heavy)
  • Consistency: inconsistent data is both valid and accurate, but which is not recorded consistently across the data according to one format.

Tidiness issues are related to problems in the structure of data which might prevent analysis. There are a few rules of thumb to follow to ensure one is working with tidy data:

  • Each variable forms a column
  • Each observation forms a row
  • Each type of observational unit forms a table

Quality¶

These are issues with accuracy, validity, consistency, & completeness

  • There were 2,075 rows in the images dataframe vs. 2,356 rows in the archive dataframe. There are several null tweet_ids
  • The timestamp column is show as an "object", while it should actually be a "datetime" object
  • Source columns values are not in a usable format for analysis with many extraneous characters
  • The most frequent "name" listed in the name column is "a" or "an", which are not actually names
  • Many dog names are entered as "none" rather than null
  • Tweet IDs are entered as both tweet_id and id - inconsistent from one dataframe to another
  • Tweet_id should be a string instead of an integer
  • "full_text" and "text" columns among dataframes contain the same info, but must be changed to the same column name of "text" for the sake of consistency
  • Archive, Images, and JSON dataframes should combined for ease of interpretation
  • Some individual entries list more than one dog stage
  • Rating Numerators and Denominators are unusual numbers - a fun quirk of this site, but difficult to analyze - both should be a float datatype

Tidiness¶

These are structural issues that should be addressed

  • The third rule of tidy data says: "each type of observational unit forms a table". The df2 and df3 are part of the same observational unit as df1 but there are three separate tables so should be merged and stored in a file called twitter_archive_master.csv, as per the project instructions.
  • Dog Stages columns have values as several columns, instead of one column filled with their values
  • Dog Breed Predictions columns have values as columns instead of one column filled with their values

Create an equation to normalize the wacky situation:
